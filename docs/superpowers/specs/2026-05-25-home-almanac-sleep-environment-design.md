# 首页黄历环境读数与睡眠设计

## 背景

HomeDeck 当前首页已经接入离线老黄历数据，`HomeRenderer` 负责 400x600 竖屏黄历渲染，`makeHomeCalendarData()` 负责把本地时间转换为首页公历与黄历字段。现有实现会在已配置启动后渲染首页，但还没有温湿度读数，也没有首页显示后自动进入 deep sleep 的运行流程。

PaperColor 板载 SHT40 温湿度传感器，内部 I2C 使用 `G2/G3`，地址为 `0x44`。Button C 对应 `G1`。ESP32-S3 SDK 支持 timer wakeup 和 deep-sleep GPIO wakeup，可用于每天午夜自动刷新和手动按键唤醒。

## 目标

- 首页左下角显示温度，格式为 `30.0°C`，距左边和底边各 12px。
- 首页右下角显示湿度，格式为 `50.0%`，距右边和底边各 12px。
- 温湿度读取失败时保留位置，显示 `--.-°C` 和 `--.-%`。
- `宜` 和 `忌` 内容最多绘制两行，超过两行的内容直接不显示。
- 已配置后的黄历首页只采样和渲染一次，显示 60 秒后进入 deep sleep。
- deep sleep 正常睡到下一个本地 `00:00:00`，用于每天自动刷新黄历。
- Button C / `G1` 可从 deep sleep 唤醒设备。
- AP 配置页不受首页睡眠逻辑影响，保持可访问。

## 非目标

- 不新增天气、历史曲线、温湿度校准、传感器配置页或报警逻辑。
- 不在 60 秒显示窗口内刷新温湿度或重新绘制首页。
- 不改变黄历数据包格式和离线黄历生成流程。
- 不改变 AP 配置页保存流程。
- 不改变 A+B 进入配置模式的现有用户意图；如果实现中发现当前代码与既定需求不一致，只在本功能范围内修正必要行为。

## 方案选择

采用方案 A：小步接入，保持现有边界。

- `HomeCalendarData` 扩展温湿度字段和可用状态。
- `HomeRenderer` 只消费数据，负责底部读数显示和 `宜/忌` 两行截断。
- `makeHomeCalendarData()` 继续只生成日期和黄历字段，不直接读硬件。
- `app_runtime` 负责读取 SHT40，并把环境读数传入首页渲染。
- `BootController` 负责系统模式下的 60 秒显示窗口和 deep sleep 触发。

这个方案避免为单次 SHT40 读数过早引入 `EnvironmentService` / `SleepService`，同时仍保持硬件读取、渲染和启动状态机的职责清晰。

## 数据与渲染

`HomeCalendarData` 增加以下字段：

- `bool temperatureAvailable`
- `float temperatureCelsius`
- `bool humidityAvailable`
- `float humidityPercent`

渲染规则：

1. 使用现有黄历主题色：周末红色、工作日绿色。
2. 使用现有设备字体，不新增字体资源。
3. 温度文本左对齐放在左下角，`x = 12`，底部距屏幕边缘 12px。
4. 湿度文本右对齐放在右下角，`x = 388`，底部距屏幕边缘 12px。
5. 可用读数保留一位小数；不可用读数显示占位。
6. `宜/忌` 仍按现有 UTF-8 code point 和字体宽度换行，但绘制最多两行。
7. `宜/忌` 行高按最多两行稳定计算，避免长文本把底部温湿度挤出屏幕。

## 传感器读取

首页渲染前只读取一次 SHT40。实现默认不新增第三方库，通过 `M5.In_I2C` 对 `0x44` 执行 SHT40 高精度测量命令，并按 Sensirion SHT4x 数据手册公式转换温度和相对湿度。

SHT40 读取对渲染层暴露的结果只有两种：

- 读取成功：温度摄氏度和相对湿度百分比。
- 读取失败：两个可用状态均为 false，首页显示占位。

传感器读取失败不阻塞黄历渲染，也不阻止后续 deep sleep。

## 睡眠与唤醒

睡眠逻辑只在 `BootMode::System` 生效。`BootMode::Config` 继续只处理 AP 配置页请求，不进入首页睡眠流程。

系统模式启动流程：

1. 恢复时区和 RTC 系统时间。
2. 读取 SHT40 一次。
3. 渲染黄历首页一次。
4. 进入 60 秒显示窗口。
5. 60 秒内不重新读取传感器，不重新渲染首页。
6. 60 秒到达后进入 deep sleep。

60 秒窗口内仍保留 A+B 配置模式重入检查。如果 A+B 触发配置重启，配置重启优先于 sleep。

进入 deep sleep 前配置两个唤醒源：

- Timer wakeup：睡到下一个本地 `00:00:00`。
- GPIO wakeup：Button C / `G1`，配置为 `INPUT_PULLUP` + 低电平唤醒。

如果当前时间不可信，无法计算下一个本地午夜，则 timer wakeup 使用 1 小时兜底。Button C 始终配置为唤醒源。

从 timer 或 Button C 唤醒后，设备按正常启动流程重新运行：恢复时间、读取温湿度一次、渲染一次、显示 60 秒、再次进入 deep sleep。

## 时间可信度

时间可信度用于决定是否能计算下一个本地午夜。

实现时以简单、可测的规则为准：

- 如果本地 Unix 时间明显早于当前项目可接受的现代时间阈值，则视为不可信。
- 如果 `std::localtime()` 返回空或得到的日期字段无效，则视为不可信。
- 不可信时不尝试猜测日期，只使用 1 小时兜底睡眠。

阈值沿用现有 NTP 同步中使用的现代时间判断口径，避免新增一套互相矛盾的时间定义。

## 测试策略

### 渲染测试

扩展 `test_home_renderer`：

- 读数可用时绘制 `30.0°C` 和 `50.0%`。
- 读数不可用时绘制 `--.-°C` 和 `--.-%`。
- 温度位于左下 12px 边距，湿度位于右下 12px 边距。
- 底部读数颜色使用当前黄历主题色。
- 长 `宜` 和长 `忌` 最多绘制两行。
- 长 `宜/忌` 不改变底部温湿度的位置。

### 启动与睡眠测试

扩展 `test_boot_controller`：

- 系统模式启动后只调用一次首页渲染。
- 60 秒内不进入 sleep。
- 60 秒到达后请求 deep sleep。
- 配置模式不会请求 sleep。
- A+B 配置重启优先于 sleep。
- 可信时间会计算到下一个本地午夜的 timer wakeup。
- 不可信时间使用 1 小时兜底 timer wakeup。
- sleep 请求包含 Button C / `G1` 低电平唤醒配置。

### Fake 支持

补强 native fake：

- `M5.BtnC` fake 观测点。
- deep sleep GPIO wakeup 配置观测点。
- `M5.In_I2C` fake 支持 SHT40 读取成功和失败两条路径。

## 验收

实现完成后的验收命令：

```bash
pio test -e native
pio run -e m5stack-papercolor
git diff --check
```

本功能不修改 LittleFS 黄历数据包，默认不要求重新生成 `buildfs`。如果实现过程中引入或调整了文件系统资源，再补跑：

```bash
pio run -e m5stack-papercolor -t buildfs
```

## 设计自检口径

- 所有新增行为都有 native test 或 fake 观测点。
- 首页 60 秒窗口内只有按键检查，没有温湿度轮询或重复渲染。
- AP 配置模式不会因为睡眠逻辑中断。
- 传感器失败只影响底部占位，不影响黄历显示和睡眠。
- 时间异常时不猜测午夜，使用 1 小时兜底。
